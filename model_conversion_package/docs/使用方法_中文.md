# 模型转换使用方法

## 1. 目标

把当前目录中的原始 checkpoint：

- `input_models/best_adaptive_net_20260316_200459.pth`
- `input_models/best_config_20260316_200459.pt`

转换成：

- `output_models/adaptive_net_android.onnx`
- `output_models/adaptive_net_android.fixed.onnx`
- `output_models/adaptive_net_android.rknn`

## 2. 文件说明

### 输入文件

- `best_adaptive_net_20260316_200459.pth`
  模型权重

- `best_config_20260316_200459.pt`
  模型配置与训练统计量

### 核心脚本

- `export_to_onnx.py`
  把 checkpoint 导出成 ONNX

- `convert_to_rknn.py`
  把固定形状 ONNX 转成 RKNN

- `torch_zip_checkpoint_loader.py`
  自定义加载器，用来正确恢复 checkpoint

## 3. 为什么不能直接用普通 torch.load

当前这份 checkpoint 在标准 `torch.load` 下，大部分大矩阵权重会被解释成接近 `1e-40` 的异常值。

这会导致：

- PyTorch 模型输出几乎不随输入变化
- ONNX 导出后的结果失真
- RKNN 转换失败或输出错误

所以这里必须用:

- `torch_zip_checkpoint_loader.py`

来参与恢复权重。

## 4. 本地方式

### 建议环境

- Python 3.11

### 安装依赖

```bash
uv python install 3.11
uv venv --python 3.11 .venv311
uv pip install --python .venv311/bin/python torch onnx onnxruntime onnxscript
```

### 导出 ONNX

```bash
./.venv311/bin/python scripts/export_to_onnx.py
```

导出后会生成：

- `output_models/adaptive_net_android.onnx`
- `output_models/adaptive_net_android.fixed.onnx`

### 转换 RKNN

注意：

`RKNN Toolkit2` 推荐在 Linux 环境中使用。为了方便交接，这里已经提供 Docker 方案，优先建议走 Docker。

## 5. Docker 方式

### 先构建镜像

```bash
./build_rknn_docker_image.sh
```

### 再执行完整转换

```bash
./run_model_conversion_in_docker.sh
```

这条命令会依次执行：

1. `export_to_onnx.py`
2. `convert_to_rknn.py`

最终输出：

- `output_models/adaptive_net_android.rknn`

## 6. 脚本执行顺序

### 路径 A：只要 ONNX

```bash
./.venv311/bin/python scripts/export_to_onnx.py
```

### 路径 B：要 RKNN

```bash
./build_rknn_docker_image.sh
./run_model_conversion_in_docker.sh
```

## 7. 输出模型说明

### adaptive_net_android.onnx

保留 batch 维动态信息的 ONNX 版本

### adaptive_net_android.fixed.onnx

固定输入形状的 ONNX 版本，输入固定为：

- `time_features`: `[1, 10, 5]`
- `scalar_features`: `[1, 2]`

### adaptive_net_android.rknn

最终可用于 Rockchip NPU 的模型文件

## 8. 成功判断标准

### ONNX 导出成功

需要满足：

- 脚本执行无报错
- 生成两个 ONNX 文件
- 模型对不同输入不是常量输出

### RKNN 转换成功

需要满足：

- `convert_to_rknn.py` 执行完成
- 成功生成 `adaptive_net_android.rknn`

## 9. 常见问题

### 问题 1：torch.load 后模型输出是常量

原因：

- checkpoint 没有被正确解释

解决：

- 不要绕过 `torch_zip_checkpoint_loader.py`

### 问题 2：RKNN 转换时报依赖错误

解决：

- 优先走 Docker 方式，不要在宿主机硬装

### 问题 3：RKNN 转换后板端仍不工作

需要区分两类问题：

1. 模型转换失败
2. 板端 RKNN runtime / HAL 本身有问题

当前这个独立包只解决第 1 类问题。
